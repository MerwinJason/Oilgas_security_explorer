Task 1 — System Understanding

Oil & Gas Facility Architecture

A smart Oil & Gas refinery operates across multiple interconnected layers. The Purdue Model (ISA-99) is the standard way to describe this — it separates the system into 5 levels, with security boundaries between them. Click any level to explore.

4
Enterprise IT Network
ERP, email, HR systems, business analytics, cloud dashboards
▶

This is the standard corporate IT environment — Windows servers, Active Directory, email, ERP (SAP), and internet-facing services. In Oil & Gas, this includes production planning software, vendor portals, and executive dashboards.

Key Assets (IT)
Active Directory servers, ERP/SAP systems, Email servers, VPN gateways, Cloud dashboards (AWS/Azure)
Data Flows
Production reports from L3 → Business KPIs, Email ↔ Internet, ERP ↔ Cloud, Remote access via VPN
Key Threats
Phishing, ransomware, credential theft, VPN exploitation, supply chain compromise
⚠ CRITICAL TRUST BOUNDARY — IT ↔ OT
3
SCADA / Operations Management
SCADA servers, historian, engineering workstations, HMI servers
▶

SCADA (Supervisory Control and Data Acquisition) provides centralized monitoring and control of the entire facility. It collects real-time data from all PLCs, stores historical records (historian), and lets operators supervise the entire plant from one interface.

Key Assets (OT)
SCADA server (Honeywell/Yokogawa/Wonderware), Process historian, Engineering workstations, HMI application servers
Data Flows
PLC status ↑ SCADA, Commands ↓ PLCs, Historical data → IT, Alarm notifications → operators
Why It's Critical
The "brain" of the plant. Compromising SCADA = full visibility over all processes and ability to issue commands to any field device.
2
Control Systems (PLCs / DCS)
PLCs, DCS controllers, Safety Instrumented Systems (SIS), local HMIs
▶

PLCs execute the actual control logic — they read sensor inputs and actuate physical devices (open/close valves, start/stop pumps, adjust pressure). In Oil & Gas, this level also includes the Safety Instrumented System (SIS), which is supposed to trigger emergency shutdowns. This is what Stuxnet targeted. This is what TRITON attacked.

Key Assets
Siemens S7 PLCs, Yokogawa DCS, Triconex SIS (safety), Local operator panels (HMIs), Remote Terminal Units (RTUs)
Protocols Used
Modbus TCP, DNP3, OPC UA, PROFIBUS, Foundation Fieldbus — most unencrypted by design
Why It's Critical
Manipulating PLC logic = physical consequences. Wrong valve position → pipeline overpressure. Disabled SIS → no emergency shutdown. This is how you cause an explosion.
⚠ OPERATIONAL BOUNDARY — Control ↔ Field
1
Basic Control Layer
Process controllers, relay logic, local control loops
▶

Automated control loops — PID controllers managing individual variables like temperature, pressure, and flow rate. Often embedded within Level 2 devices in modern plants.

Function
Closed-loop control: maintain process variable at setpoint (e.g., keep pressure at 50 PSI)
Risk
Setpoint manipulation: attacker changes target value to unsafe range
Example
Pressure controller for a pipeline section — alter the setpoint and the pipeline ruptures
0
Field Devices (IIoT / Sensors)
Temperature sensors, pressure gauges, flow meters, actuators, valves, pumps
▶

Physical hardware that interacts directly with the process. Sensors measure real-world conditions; actuators make things happen. IIoT devices here connect to cloud dashboards via edge gateways — this is the newest and often least-secured layer.

IIoT Devices
Smart sensors (temp/pressure/flow), Edge gateways, Smart valves with digital interfaces, Wireless mesh sensor networks
Data Path
Sensor → PLC (wired/wireless) → SCADA, OR Sensor → Edge Gateway → Cloud Dashboard (bypassing OT entirely)
IIoT Risk
Default credentials, no encryption, unpatched firmware, direct cloud connectivity bypassing DMZ — a common blind spot in OT security
📊 Data Flow Summary

Physical process → Sensors → PLCs (control logic) → SCADA (supervision) → IT Network (business reporting) → Cloud (remote dashboards)

⚠ Attacks typically move in reverse: Internet → IT → OT → Field devices

🔒 Trust Boundaries (Key Concept)
  • Internet → IT: Firewall, DMZ
  • IT → OT: The most critical boundary. Requires data diode or unidirectional gateway
  • OT → Field: Physical security + protocol filtering
  • Cloud ↔ OT: Often poorly secured — direct IIoT paths bypass normal boundaries

Key Insight for the Assignment

In OT, Availability and Integrity are more critical than Confidentiality — the reverse of typical IT priorities. A 30-second SCADA outage in a refinery is catastrophic. A modified PLC setpoint that nobody notices for 2 hours can kill people. This drives every design decision in defense.

Task 3 — MITRE ATT&CK for ICS

Technique Mapping — Oil & Gas

MITRE ATT&CK for ICS is a separate framework from the Enterprise version. It uses technique IDs in the T08xx range. The sample answer incorrectly used Enterprise mitigations (M1030, etc.). These are the real ICS techniques relevant to your scenario.

⚠ Common Mistake to Avoid

The sample answer used M1030 (Network Segmentation) and M1032 (MFA) — these are from MITRE ATT&CK Enterprise. The ICS framework has its own mitigations: M0810 (Network Allowlists), M0930 (Network Segmentation), etc. Using the right codes shows you know the actual ICS framework.

T0865
Spearphishing Attachment
Initial Access
How it worksAttacker sends a targeted email with a malicious attachment (weaponized PDF or XLSX) to an oil & gas engineer or IT admin. Once opened, it installs a backdoor on the corporate IT machine — the first step into the network.
Oil & Gas angleEngineers regularly receive technical documents from vendors. A fake IEC 61511 compliance report or updated P&ID drawing is highly believable. Real precedent: TRITON began with spearphishing a Saudi Aramco affiliate.
Components affectedIT Workstation Engineering PC
ICS MitigationM0949 Antivirus/Anti-malware, M0921 Restrict Web-Based Content, User awareness training
T0822
External Remote Services
Initial Access
How it worksExploitation of exposed remote access services — VPN gateways, RDP, VNC, TeamViewer — that allow external connectivity to the OT environment. Often involves weak/default credentials or unpatched vulnerabilities.
Oil & Gas angleRemote monitoring is standard in Oil & Gas (vendors, engineers on-call). Colonial Pipeline was breached via a compromised VPN account with no MFA. Oldsmar water plant via TeamViewer. These aren't unusual — they're industry norms.
Components affectedVPN Gateway SCADA Eng. Workstation
ICS MitigationM0804 Authorization Enforcement, M0927 Password Policies, MFA for all remote access
T0807
Command-Line Interface
Execution
How it worksAfter gaining access to a workstation or SCADA server, the attacker uses command-line tools (cmd, PowerShell, bash) to execute reconnaissance commands, install malware, or interact with industrial software interfaces.
Oil & Gas angleSCADA engineering workstations run Windows. Once an attacker has a shell, they can run commands like net scan, access PLC programming software (Step 7, RSLogix), and use vendor CLI tools to talk directly to PLCs over Modbus.
Components affectedSCADA Server Eng. Workstation
ICS MitigationM0938 Execution Prevention, application allowlisting, disable unnecessary CLIs on OT hosts
T0889
Modify Program
Persistence
How it worksAttacker modifies the logic program running on a PLC or RTU. The modified program includes a backdoor or altered process behavior. Since PLCs retain their programs across reboots, this achieves persistence automatically.
Oil & Gas angleThis is the core Stuxnet technique. In a pipeline context: modify the valve control logic so that a specific valve receives an "open" command when it should close, triggered at a preset time or condition. Operator display still shows "closed."
Components affectedPLC RTU
ICS MitigationM0802 Communication Authenticity, M0947 Audit, integrity monitoring of PLC programs (hash comparison)
T0857
System Firmware
Persistence
How it worksAttacker modifies the firmware of an ICS device (PLC, RTU, sensor gateway). Firmware modifications are extremely persistent — they survive factory resets and firmware updates depending on implementation.
Oil & Gas angleIndustrial PLCs and smart sensors often allow firmware uploads through their programming software. A compromised engineering workstation could push malicious firmware to multiple PLCs in a single automated session, affecting an entire pipeline segment.
Components affectedPLC IIoT Sensors Edge Gateways
ICS MitigationM0945 Code Signing, M0946 Boot Integrity, vendor firmware authentication requirements
T0812
Default Credentials
Lateral Movement
How it worksExploiting unchanged factory-default credentials on OT devices to gain unauthorized access. Extremely common in ICS environments where devices are deployed and never have credentials changed.
Oil & Gas angleSiemens SIMATIC PLCs, Honeywell DCS units, and many SCADA historian databases have well-known default passwords. Shodan regularly indexes exposed ICS devices with default creds. APT groups compile these databases. Once in the OT network, this is the fastest lateral movement path.
Components affectedPLC SCADA HMI
ICS MitigationM0927 Password Policies — mandatory credential rotation; asset management registry to track all default credential instances
T0843
Program Download
Lateral Movement
How it worksUsing legitimate OT protocols (Modbus, EtherNet/IP, PROFINET) to download modified programs to PLCs from a compromised engineering workstation. This is legitimate functionality being abused — standard engineering practice, no exploit required.
Oil & Gas angleEngineers routinely push logic updates to PLCs from workstations. An attacker with workstation access uses the exact same workflow — Step 7 software, OPC UA connection — to push malicious programs to 15 pipeline PLCs in 10 minutes. Operators see nothing unusual because the process looks identical.
Components affectedPLC Eng. Workstation
ICS MitigationM0802 Communication Authenticity, M0813 Software Process and Device Authentication — only authenticated workstations should be able to push PLC programs
T0840
Network Connection Enumeration
Discovery
How it worksScanning the OT network to discover connected devices, open ports, and running protocols. In ICS environments, passive scanning of Modbus, DNP3, and EtherNet/IP traffic is sufficient — devices announce themselves constantly.
Oil & Gas angleOnce inside the OT network segment, an attacker runs passive ICS-aware scanners (Claroty, Dragos equivalent tools) to map every PLC, RTU, sensor, and their firmware versions within hours. This is the recon phase before targeted manipulation. Flat OT networks make this trivially easy.
Components affectedOT Network All OT Devices
ICS MitigationM0930 Network Segmentation — micro-segmentation limits blast radius; OT network monitoring (Dragos/Claroty) detects unusual scanning behavior
T0856
Spoof Reporting Message
Inhibit Response Function
How it worksAttacker injects false status messages into the SCADA communication stream so that operators see normal readings while the actual process is compromised. Stuxnet's defining technique: centrifuges were destroying themselves while the control room showed "all systems normal."
Oil & Gas anglePipeline pressure is climbing dangerously. The SCADA display shows 60 PSI (normal) because the attacker has injected spoofed Modbus response packets. Operators have no idea. Safety alarms don't trigger. This is the most dangerous inhibit technique because it creates a window of time for physical damage to build undetected.
Components affectedSCADA Operator HMI
ICS MitigationM0802 Communication Authenticity, independent physical gauges, cross-validation between SCADA and physical sensors
T0838
Modify Alarm Settings
Inhibit Response Function
How it worksAttacker accesses SCADA configuration and raises alarm thresholds or disables specific alarms entirely. Operators continue working, unaware that safety thresholds have been silently modified.
Oil & Gas angleChange the high-pressure alarm threshold from 80 PSI to 200 PSI in SCADA config. Now the pipeline can reach dangerous levels without triggering any alert. Operators are trained to respond to alarms — if there's no alarm, they assume everything is fine.
Components affectedSCADA DCS
ICS MitigationM0947 Audit — automated integrity checking of alarm configurations; change management process for all SCADA settings
T0836
Modify Parameter
Impair Process Control
How it worksDirect modification of process control parameters — setpoints, gain values, limit values — via OT protocols. No program modification required; just write a new value to the PLC register. Modbus write commands require no authentication by default.
Oil & Gas angleWrite a new setpoint to the PLC register controlling the pressure relief valve: change from "open at 90 PSI" to "open at 500 PSI." The pipeline now has no effective pressure relief. When pressure builds, no physical safety response occurs. This is a direct path to pipeline rupture.
Components affectedPLC DCS SIS
ICS MitigationM0800 Access Management, M0801 Access Control — restrict which devices can write to PLC registers; use protocol-aware firewalls to block unauthenticated Modbus write commands
T0880
Loss of Safety
Impact
How it worksThe ultimate ICS attack goal: compromise the Safety Instrumented System (SIS) so that it cannot perform its protective function. TRITON/TRISIS is the only known malware designed specifically to do this.
Oil & Gas angleThe SIS (e.g., Triconex) is the last line of defense — it triggers emergency shutdown when pressures, temperatures, or flows exceed safe limits. TRITON put the SIS into a compromised state: it would either fail to trigger a shutdown, or trigger one at the wrong time. In an Oil & Gas facility, this is the difference between a near-miss and a mass casualty event.
Components affectedSIS (Triconex) Safety Controllers
ICS MitigationM0810 Network Allowlists, physical isolation of SIS network, diverse safety system (independent hardware, different vendor from main control system)
T0813
Denial of Control
Impact
How it worksPreventing operators from issuing control commands to field devices. Can be achieved by DoS on SCADA/OT network, locking operators out of HMI accounts, or triggering safe-state modes on PLCs that block remote commands.
Oil & Gas angleDuring an emerging pipeline leak, operators lose the ability to close emergency shutoff valves through SCADA. They must physically dispatch field teams to every valve — potentially taking hours while the leak continues. This combines physical disruption with maximized environmental and safety damage.
Components affectedSCADA PLC Operator HMI
ICS MitigationM0810 Network Allowlists, M0953 Data Backup — manual override procedures for all critical valves; backup communication paths (radio, serial)
Task 4 — Adversary Simulation

APT Attack Chain — Pipeline Control

A realistic APT attack against an Oil & Gas facility inspired by TRITON (2017) and Industroyer (2016). Unlike Stuxnet which targeted a specific facility, this is a multi-stage campaign designed to achieve maximum physical impact. Click each step to expand.

🎯 Objective

Sabotage a liquefied natural gas (LNG) pipeline facility — cause overpressure event leading to rupture, while disabling the SIS to prevent automatic emergency shutdown. Maximize damage while maintaining covert presence as long as possible.

👤 Threat Actor Profile

Nation-state APT group (comparable to XENOTIME, the TRITON actors). Motivated by geopolitical disruption. Has OT-specific technical capability. Prepared to spend 6-18 months in reconnaissance before triggering the payload.

1
Initial Compromise — Spearphishing
T0865 · Initial Access · IT Network

The APT group researches the target over 3 months using OSINT — LinkedIn profiles of SCADA engineers, company vendor relationships, recent regulatory filings that mention their ICS platform (Honeywell Experion PKS).

Attacker Action

Sends a targeted email to the SCADA engineer. Subject: "Q3 Process Safety Review — IEC 61511 Compliance Template." Attachment is a weaponized Excel file with embedded macro. Engineer opens it, thinking it's routine.

Detection Opportunity

Email security gateway scanning attachments; macro execution alert on endpoint; EDR detecting unusual Excel process spawning PowerShell. Most OT-connected IT systems lack these controls.

2
IT Persistence — Backdoor Installation
T0807 · Execution · IT Workstation

The macro installs a RAT (Remote Access Trojan) that beacons over HTTPS to a C2 server. The attacker now has persistent access to the engineer's workstation — in the enterprise IT network.

Attacker Action

RAT runs as a scheduled task mimicking a legitimate Windows Update service. It exfiltrates Active Directory credentials, network topology maps, and VPN configuration files over 4 weeks — slow and quiet to avoid anomaly detection.

Detection Opportunity

SIEM correlation of unusual outbound HTTPS to unknown domains; User and Entity Behavior Analytics (UEBA) flagging credential access outside normal hours; DLP alert on network diagram exfiltration.

3
IT → OT Lateral Movement
T0822 · Lateral Movement · DMZ Crossing

Using stolen credentials, the attacker accesses the engineering jump server — a machine that sits between IT and OT networks. This is the critical IT-OT boundary crossing.

Attacker Action

Logs into the VPN using stolen credentials. Pivots to the jump server. From there, enumerates the OT network segment using passive listening (Wireshark on Modbus/OPC UA traffic). Maps every PLC, their IP addresses, firmware versions, and engineering software in use.

Detection Opportunity

MFA on VPN would block this immediately. Anomaly detection on jump server login — unusual time, unusual source IP. OT network monitoring (Dragos/Claroty) detecting passive scanning patterns.

4
SCADA Access — Command & Control Established
T0812 · Lateral Movement · SCADA Server

The SCADA historian server is accessible from the jump server using default credentials that were never changed. The attacker now has full visibility into all process data and can interact with the SCADA application.

Attacker Action

Installs a second backdoor on the SCADA server for redundant access. Downloads the engineering project files — these contain the full PLC program logic, I/O map, setpoints, and alarm configurations for the entire facility. This is the "blueprint" for the attack.

Detection Opportunity

Changed default credentials would have prevented this. File access audit logs on SCADA server. DLP alert on engineering project file download. Application allowlisting preventing unauthorized software installation.

5
SIS Attack — Safety System Compromised (TRITON-style)
T0880 · Impact · Safety Instrumented System

This is the TRITON technique. The SIS (Triconex controller) is accessed via the engineering network. The attacker reprograms the SIS to either fail-safe in an unsafe state or to not respond to shutdown triggers.

Attacker Action

Uses TriStation protocol (the Triconex engineering protocol) to inject modified SIS logic. The new logic: when pressure exceeds 120 PSI (threshold for emergency shutdown), instead of triggering ESD, the SIS enters a "hold" state. All SIS diagnostic lights continue showing green.

Detection Opportunity

Physical isolation of SIS network from control network. TRITON was actually discovered accidentally when a programming error caused the SIS to initiate a safe shutdown, alerting engineers. SIS integrity checking — compare running program to known-good backup.

6
Process Manipulation — Pipeline Overpressure
T0836 + T0856 · Impair Process + Inhibit Response

With the SIS compromised and the ability to spoof SCADA readings, the attacker triggers the physical attack. PLC setpoints are modified via T0836, while T0856 ensures operators see nothing unusual.

Attacker Action

Writes new pressure setpoints to the pipeline section PLCs via Modbus — closing downstream pressure relief valves while maintaining upstream pump pressure. Simultaneously injects spoofed Modbus response packets so SCADA reads 60 PSI (normal). Physical pressure climbs to 180+ PSI undetected. Pipeline ruptures.

What Should Have Stopped This

Independent physical pressure gauges not connected to SCADA. Protocol-aware firewall blocking unauthenticated Modbus writes. Redundant SIS with different communication path. Anomaly detection on PLC I/O behavior. Real-time cross-validation between SCADA and field instruments.

Why This Differs From the Sample Answer

The sample answer has 7 generic steps with no technical depth. A strong answer: (1) uses real ICS-specific techniques with T-codes, (2) shows how each step works technically, (3) explains what makes detection hard at each stage, (4) draws on real incidents (TRITON specifically is ideal for Oil & Gas), and (5) connects the attack steps to specific components in the architecture.

Task 5 — Defense Strategy Design

Multi-Layered Security Architecture

The "defense-in-depth" principle: assume the attacker will breach the outer layer, so build independent defenses at every layer. Click each layer to expand controls and their MITRE ICS mitigation mappings.

Layer 1 — Perimeter & Network Segmentation
Preventive · Keep attackers from crossing IT→OT boundary
▾

The most impactful single control for OT security. Without proper segmentation, once an attacker breaches IT, they have unrestricted access to every PLC in the facility.

IT/OT Firewall + DMZ Unidirectional Gateway (data diode) OT VLAN segmentation SIS network isolation (air-gap) Protocol-aware firewall (blocks Modbus writes from IT)

MITRE Mitigations: M0930 Network Segmentation · M0810 Network Allowlists

Layer 2 — Identity & Access Control
Preventive · Zero Trust — never trust, always verify
▸

Default credentials and shared accounts are responsible for a large percentage of ICS breaches. Every device must have unique credentials; every remote access must use MFA.

MFA on all remote access (VPN, RDP, jump server) Change all default OT device credentials Role-based access control (RBAC) on SCADA Privileged Access Workstations (PAW) for OT engineers Disable unused services/ports on PLCs Zero Trust Architecture for SCADA access

Layer 3 — OT Monitoring & Anomaly Detection
Detective · Know what "normal" looks like — detect deviations
▸

OT-aware IDS (Dragos Platform / Claroty) SCADA anomaly detection (unusual commands) PLC integrity monitoring (hash of program) SIS diagnostic monitoring (independent channel) SIEM integration (OT + IT log correlation) AI-based process anomaly detection

Task 6 — Critical Evaluation

Limitations, Challenges & Gaps

This is the most intellectually demanding task. It asks you to think critically about your own work — what does MITRE ATT&CK for ICS miss? What can't you actually defend against in a real refinery? Being honest here demonstrates deeper understanding than a confident-sounding but shallow answer.

Limitations of MITRE ATT&CK for ICS

⚡ No Process Safety Context

ATT&CK for ICS describes attack techniques but doesn't model the consequence severity. T0836 (Modify Parameter) applied to a pump setpoint vs. a pipeline pressure relief valve have radically different consequences, but ATT&CK treats them identically. It's not a risk model — it's a technique library. In Oil & Gas, consequence modeling (HAZOP, SIL analysis) is a separate discipline that doesn't integrate with ATT&CK.

📊 Incomplete Coverage of OT-Specific Protocols

MITRE ATT&CK for ICS doesn't have dedicated techniques for attacks on specific industrial protocols like DNP3 authentication bypass, PROFIBUS timing attacks, or OPC DA vulnerabilities. Researchers have demonstrated attacks on these protocols that don't map cleanly to existing ATT&CK techniques. The framework lags behind the actual threat research in ICS security.

🔄 Biased Toward Attack Simulation, Not Defense Design

ATT&CK was built for red teams and threat intelligence — it describes what attackers do. The mitigation mappings (M-codes) are high-level and don't provide practical implementation guidance for OT environments. A SCADA engineer reading M0930 (Network Segmentation) has no guidance on how to segment a live refinery without causing operational disruptions.

🏭 Assumes IT-Style Logging and Visibility

Many ATT&CK detection strategies assume endpoint logging, SIEM, and network traffic visibility. Legacy PLCs from the 1990s-2000s produce no logs whatsoever. A Siemens S5 PLC doesn't have a syslog output. Detection strategies based on log analysis are simply inapplicable to a large portion of Oil & Gas installed base.

Challenges in Oil & Gas OT Security

⏰ Legacy Systems & Operational Continuity

A refinery operating 24/7 cannot simply take a PLC offline to apply security patches. The maintenance windows are planned 6-18 months in advance and last hours, not days. Security patches that require reboots compete directly with production continuity. In practice, many OT systems remain unpatched for years — not out of negligence, but structural constraint.

⚡ Real-Time Constraints vs. Security Controls

Deep packet inspection, encryption, and certificate-based authentication all add latency. For SCADA-to-PLC communications controlling a pressure relief system, a 50ms delay introduced by a TLS handshake could be the difference between safe operation and an over-pressure event. Many encryption and authentication controls are genuinely incompatible with hard real-time OT requirements.

💰 Cost and OT Security Expertise Gap

OT security specialists who understand both ICS protocols and cybersecurity are rare and expensive. Many refineries rely on the same engineers for both operations and security — people trained in chemical engineering and process control, not network security. TRITON was missed for months partly because nobody was specifically looking for SIS-targeting attacks.

🔗 Supply Chain and Vendor Risk

OT vendors regularly need remote or on-site access for maintenance. Each vendor connection is a potential entry point — and the facility has limited visibility into the security posture of third-party vendors. Contractual security requirements for OT vendors are often weak or unenforceable in practice.

Gaps in the Security Design (Honest Self-Evaluation)

⚠ The IIoT Problem Remains Largely Unsolved

The defense strategy focuses heavily on IT/OT boundary and SCADA-to-PLC communications. IIoT sensors with direct cloud connectivity bypass the entire OT security architecture. Many facilities add IIoT layers without integrating them into the existing security framework, creating a parallel unmonitored attack surface. Our defense design doesn't adequately address this.

👤 Insider Threat Detection is Weak

A legitimate engineer with authorized SCADA access and knowledge of the PLC programs is nearly indistinguishable from an attacker who has compromised that engineer's credentials. UEBA tools can detect statistical anomalies, but a patient, knowledge-aware insider (or a compromised insider account used carefully) is the hardest threat to detect with technical controls.

What Makes Task 6 Stand Out

Most students will write "MITRE ATT&CK has limitations in OT environments" and list three generic points. What separates a strong answer: specificity. Name the actual limitation, explain why it's a limitation in the ICS context specifically (not just generally), and — where possible — reference how it played out in a real incident. The self-evaluation of your security design gaps shows you understand the difference between a security architecture on paper and one that actually works in a live refinery.
